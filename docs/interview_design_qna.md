# AI Agent Learning Coach 面试设计问答

问题：这个项目的整体定位是什么？

答案：这是一个面向 AI Agent 学习者的智能学习教练 App。用户注册登录后输入学习目标，系统会生成学习路线；用户点击某一周后，后端通过 LangGraph 调度多个 Agent，依次生成本周学习内容、GitHub 开源项目推荐、作品集实战任务、测验题和模拟面试题。它不是单纯聊天机器人，而是一个有学习闭环的多智能体应用。

问题：为什么一开始要先分析参考项目架构？

答案：因为用户最初对项目组成没有整体观。先分析 `helloagents-deepresearch` 这类参考项目，可以明确哪些能力值得借用：任务拆解、搜索总结、多 Agent 编排、流式输出、结构化结果、资料引用。之后再结合 `more agents` 的定位，把它收敛成“学习路线 + 开源分析 + 作品集任务 + 测评面试”的主链路，避免功能堆砌。

问题：为什么把 OpenSource Mentor Agent 和 Project Coach Agent 合并？

答案：这两个 Agent 的输入和目标高度相关。OpenSource Mentor 搜索并分析 GitHub 项目，Project Coach 基于这些项目生成作品集级任务。如果拆得太细，状态传递更复杂；合并后可以在同一个节点里完成“搜索开源项目 -> 分析仓库 -> 生成实战任务”，减少链路长度，也更符合用户点击某周后一次性得到可执行内容的体验。

问题：为什么选择 FastAPI + Vue + TypeScript + SSE？

答案：FastAPI 适合构建异步 API 和流式响应，后端可以承载 LangGraph 工作流。Vue + TypeScript 适合快速构建结构化前端，并通过类型约束减少接口字段错用。SSE 适合 Agent 逐步生成内容的场景，服务端可以不断推送节点进度和中间状态，前端不用等整个图跑完才显示结果。

问题：为什么前后端端口不一样也能通信？

答案：前端 Vite dev server 和后端 FastAPI 是两个独立服务，端口不同很正常。开发环境里前端可以通过 `VITE_API_BASE` 或 Vite proxy 把请求转发到后端端口，例如前端在 5173，后端在 8001。浏览器看到的是前端页面，但 API 请求会被转到 FastAPI。

问题：为什么要移除 Streamlit 和 mock？

答案：项目已经转向真实的前后端应用，Streamlit 更适合原型展示，不适合后续做登录、JWT、用户隔离、SSE、复杂交互状态。mock 会掩盖真实链路问题，比如数据库、GitHub Token、SSE 鉴权、Agent 输出结构等，所以先移除 mock，保证完整功能都跑真实后端。

问题：为什么使用 LangGraph 架构？

答案：这个项目不是一次性问答，而是多阶段工作流。LangGraph 可以把 Planner、Tutor、OpenSource Mentor、Reporter、Assessment、Interview 等 Agent 组织成节点，通过 Supervisor 根据 state 决定下一步。相比普通顺序 Chain，LangGraph 更适合表达状态、条件路由、循环补救和长流程追踪。

问题：主链路是怎么设计的？

答案：主链路是：用户登录 -> 输入学习目标 -> Planner 生成学习路线 -> 用户点击某一周 -> Tutor 生成学习内容 -> OpenSource Mentor 搜索 GitHub 并生成作品集任务 -> Reporter 生成学习报告 -> Assessment 生成选择题 -> 用户提交测验 -> 分数通过进入 Interview -> 用户提交面试答案 -> 系统评分并给参考答案 -> 完成本周进入下一周。

问题：为什么刚提出学习目标后只显示学习路线？

答案：这是为了降低首屏信息噪音。用户刚输入目标时，最需要的是确认学习路线是否合理；GitHub 推荐、作品集任务、测验和面试都依赖具体周主题。如果一开始全部生成，会慢、乱、也难以解释。因此先显示路线，用户点击第一周后再按周生成内容。

问题：为什么点击“开始本周”后采用流式输出？

答案：因为一周内容涉及多个 Agent，GitHub 搜索和 LLM 生成都可能耗时。如果等全部完成再返回，用户会以为页面卡死。SSE 可以把每个节点完成后的 state 推给前端，用户能逐步看到学习内容、开源推荐、项目任务、测验等模块出现，体验更像一个真实工作的 Agent 系统。

问题：为什么要做 JWT 鉴权？

答案：项目涉及用户学习路线、测验结果、面试答案和长期记忆，这些都属于用户私有数据。JWT 可以让前端每次请求都携带身份，后端通过 token 解析出当前用户，再按 `user_id` 查询和保存数据。这样避免只靠 `learner_id` 字符串导致串号或越权访问。

问题：数据库是如何按用户隔离的？

答案：核心表如 `coach_sessions`、`coach_memories` 都增加了 `user_id`。后端接口通过 `get_current_user` 从 JWT 中拿到当前用户，再调用存储层时传入 `current_user.id`。读取 session 时会同时过滤 `session_id` 和 `user_id`，如果不是当前用户的数据，就返回不存在。

问题：为什么用户注册时曾经出现“请求的数据不存在，可能已被删除或不属于当前用户”？

答案：这是因为注册和登录/恢复会话的错误处理混在了一起。注册本身不应该读取学习 session，但前端或错误归一化逻辑把 404 会话错误展示到了注册流程里。解决方法是把认证错误和业务数据错误分开处理，注册只处理用户名、密码、用户已存在等认证相关错误。

问题：为什么同一个账号刷新后之前的内容消失？

答案：后端其实已经把 session 保存到了数据库，但前端刷新后只保留了 token，没有恢复上次 `session_id` 和 state。解决方法是新增 `GET /learning-coach/sessions/latest` 接口，按当前 `user_id` 查询最近一次会话；前端刷新或登录成功后自动调用这个接口恢复上次内容。

问题：为什么还要加“恢复上次记录”按钮？

答案：自动恢复可以解决刷新丢状态的问题，但用户也需要一个明确入口手动拉回历史内容。这样“恢复上次记录”表示读取数据库，不重新跑 Agent；“重新生成学习路线”才表示新建会话、重新规划、重新搜索 GitHub。两个动作语义清楚，避免误触发昂贵的搜索和生成。

问题：为什么要把数据同时归档到本地文件？

答案：PostgreSQL 是主存储，但开发环境中数据库可能配置错误、断开或迁移失败。文件归档作为 fallback，可以在数据库不可用时保留 session JSON，避免用户生成的内容完全丢失。后来还补了 `list_json` 能力，让最近会话列表在数据库不可用时也能从本地归档恢复。

问题：GitHub Token 是什么意思？

答案：GitHub Token 是访问 GitHub API 的凭证。没有 Token 时也可以访问公开接口，但搜索频率限制低，容易失败。有 Token 后可以更稳定地搜索仓库、读取 README 和获取 star、语言、更新时间等元数据。这个 Token 应该放在 `.env` 中，不能提交到 Git 仓库。

问题：为什么 GitHub 推荐有时为空？

答案：可能原因包括：没有配置 GitHub Token、网络或 API 限制、搜索关键词太窄、当前学习目标没有英文关键词、GitHub API 返回失败。解决时做了两层处理：一是根据学习目标和当前周主题生成搜索关键词；二是当搜索失败时使用内置参考仓库作为 fallback，并在页面标记“内置参考”。

问题：为什么 GitHub 推荐要跟学习目标里的关键词绑定？

答案：如果固定列出 `langgraph`、`langchain`、`fastapi` 这些仓库，就像静态推荐，不能体现 Agent 的个性化能力。把搜索关键词来自用户学习目标和当前周主题，可以让 GitHub 推荐随用户目标变化。例如用户学 MCP，就优先搜 MCP 相关仓库；用户学 LangGraph，就优先搜图编排相关仓库。

问题：为什么选择题未通过后不能直接进入下一周？

答案：学习产品需要闭环，不应该让用户跳过薄弱点。Assessment 分数低于 60 时，Supervisor 会把流程路由回 Tutor，生成补救学习内容；前端也禁用“完成本周并继续”，要求用户复习薄弱点并重新测验。这样保证每周学习有最低掌握标准。

问题：为什么低分后重新测验要清空红绿答案？

答案：低分后如果红绿正确/错误状态还保留，用户会看到答案提示，重新作答就失去测评意义。解决方法是把 `Retry Quiz` 做成重新作答模式：清空本地选择、隐藏 `score`、去掉 `correct_answer/user_answer/is_correct` 展示状态，再让用户重新提交。

问题：为什么答案和解析要中文？

答案：项目定位是中文学习 App，用户学习、测评和面试复盘都应该用中文。之前只有部分 UI 中文，Agent 输出仍可能是英文，体验割裂。解决方法是在 Assessment、Tutor、Interview 和面试评分 prompt 中明确要求中文输出，并给选择题增加 `explanation` 字段，提交后展示“正确答案”和“解析”。

问题：为什么面试部分要先让用户输入答案，再评分和显示参考答案？

答案：如果一开始就显示参考答案，用户只是阅读而不是作答，无法形成面试训练。现在的流程是：先生成问题 -> 用户输入答案 -> 后端根据参考答案评分 -> 显示反馈、亮点、改进建议 -> 用户点击按钮查看参考答案。这样更接近真实面试，也能检验表达能力。

问题：为什么切到下一周后，面试反馈曾经显示上一周内容？

答案：切换周时后端只清空了 `interview_questions`，但没有清空 `interview_answers`、`interview_evaluations` 和 `interview_score`。前端看到旧评分字段，就误以为新周面试已经提交过。解决方法是在 `_prepare_week_state()`、初始 state 和重新提交测验时统一清空面试相关状态。

问题：为什么要把页面模块做成可折叠？

答案：一周内容包含学习路线、学习内容、GitHub 推荐、作品集任务、报告、测验、面试，信息量很大。如果全部展开，页面会很长且难以定位。可折叠模块让用户按当前任务聚焦，例如先看学习内容，再展开测验，最后展开面试反馈。

问题：为什么要做完整错误反馈？

答案：Agent 应用失败点很多，包括后端未启动、端口占用、Token 过期、数据库不可用、GitHub API 失败、SSE 中断、当前用户无权访问数据等。如果只显示 `Internal Server Error`，用户无法判断下一步。解决方法是在后端返回明确中文错误，在前端按 HTTP 状态码和业务 detail 统一展示。

问题：为什么 SSE 需要把 token 放进 query？

答案：浏览器原生 `EventSource` 不支持像 `fetch` 那样自定义 Authorization header。为了让 SSE 请求也能鉴权，只能把 access token 放到 query 参数里，后端通过依赖解析当前用户。这个方案实现简单，但生产环境要注意 HTTPS、短期 token 和日志脱敏。

问题：为什么端口被占用会导致服务启动失败？

答案：同一台机器同一个端口只能被一个进程监听。之前 FastAPI 启动时报 `WinError 10048`，说明 8001 已经被旧进程占用。解决方法是关闭旧进程，或者修改 `.env` 中的 `PORT` 换一个端口，并同步前端代理或 `VITE_API_BASE`。

问题：为什么模型名可以配置？

答案：项目通过环境变量配置不同 Agent 使用的 LLM，例如 `COACH_LLM_MODEL=deepseek-chat`。这样不用改代码就能切换模型。是否能用某个模型名取决于服务商是否支持，例如不存在或无权限的模型名会导致调用失败，所以需要按 API 平台实际模型列表配置。

问题：这个项目目前是真 MCP 吗？

答案：目前更准确地说是 MCP 风格的工具适配层，不是完整 MCP Server。项目里有 `github_client.py` 和 `filesystem_client.py` 这类适配器，提供类似 MCP 工具边界的封装，但还没有实现完整的 MCP 协议服务端、工具发现和标准传输。面试时可以说：当前阶段先实现 MCP Adapter，后续可升级为真正 MCP Server。

问题：为什么要把页面从 Markdown 输出改成结构化输出？

答案：Markdown 虽然生成简单，但前端难以做稳定排版和交互。结构化 JSON 例如 `tutor_task`、`project_task`、`quiz`、`interview_evaluations` 能让前端用固定模块展示标题、目标、概念、步骤、链接、评分等内容，也方便做折叠、按钮、状态和样式控制。

问题：为什么学习视频链接目前是搜索链接，而不是直接视频？

答案：直接推荐具体视频需要实时搜索、筛选质量和处理失效链接，成本更高。当前先生成稳定搜索链接，例如 YouTube、Bilibili、GitHub 搜索，保证任何主题都能打开并找到相关资料。后续可以升级为真正的视频搜索 Agent，返回标题、作者、时长、链接和推荐理由。

问题：Portfolio Task 做出来以后怎么办？

答案：更合理的下一步不是直接帮用户提交 GitHub，而是做“作品提交与反馈闭环”。用户提交 GitHub Repo URL、Demo 链接和 README，Portfolio Reviewer Agent 分析代码结构、README、功能完成度和测试情况，然后输出评分、改进建议、简历描述和面试讲法。这样能把学习项目变成真正可展示的作品集成果。

问题：为什么不直接做 GitHub 提交按钮？

答案：直接向用户 GitHub 仓库提交代码需要 OAuth、权限授权、分支管理、冲突处理和安全审计，复杂度较高。当前更适合先做“提交项目链接并评审”，等项目评审闭环稳定后，再考虑 GitHub OAuth 和自动 PR。

问题：这个项目面试时最应该强调什么？

答案：最应该强调它不是一个简单 LLM 包装，而是一个完整的 Agent 产品闭环。它包含 LangGraph 多节点编排、JWT 鉴权、用户数据隔离、PostgreSQL 持久化、SSE 流式输出、GitHub 工具适配、结构化前端渲染、测验补救闭环、面试评分闭环和历史会话恢复。这些设计都围绕真实用户学习过程，而不是为了堆技术名词。
